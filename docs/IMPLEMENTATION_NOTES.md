# 実装・設計判断の記録

このドキュメントには、機能追加や不具合修正で行った変更と、その判断理由を記録します。
コードだけでは分かりにくい設計意図を、後から確認できる状態にすることが目的です。

## 記録方針

実装や修正を行う際は、次の内容を基本として記録します。

- 何を変更したか
- なぜ変更したか
- なぜその方法を選んだか
- 他に検討した方法
- 確認方法と結果

小さな文言修正や自動整形など、設計判断を伴わない変更は関連する記録へまとめる場合があります。

## 記録テンプレート

```md
## YYYY-MM-DD: 変更内容のタイトル

### 何を変更したか

変更した機能やファイルを簡潔に記載します。

### なぜ変更したか

利用者が困っていたことや、解決したい問題を記載します。

### なぜこの方法を選んだか

既存設計との関係、変更範囲、保守性などの判断理由を記載します。

### 他に検討した方法

採用しなかった方法と、その理由を簡潔に記載します。

### 確認

実行したビルド、テスト、画面確認などを記載します。
```

## 2026-06-22: サイドバーを編集エリアの高さまで伸ばす

### 何を変更したか

- `app/page.tsx` のサイドバー外側要素を、PC表示時にFlexコンテナとして縦方向へ伸ばすよう変更
- `components/Sidebar.tsx` と `components/TodoSidebar.tsx` が、外側要素の高さを使えるよう調整
- メモとToDoの両方へ同じ指定を適用

### なぜ変更したか

編集エリアが画面より高くなった場合、サイドバーだけが先に終わり、ページを下へスクロールすると背景や下部UIが途中で見切れていました。
サイドバー内の項目数が少ないときに特に目立つ問題でした。

### なぜこの方法を選んだか

画面全体のスクロール方法や編集エリアの高さは変更せず、サイドバーを包む要素だけを編集エリアと同じ高さへ伸ばす方法を選びました。
既存レイアウトへの影響が小さく、メモとToDoへ同じ仕組みを適用できるためです。

### 他に検討した方法

- サイドバーを `position: sticky` または `fixed` にする
  - スクロール挙動が大きく変わり、設定メニューやモバイル表示への影響があるため不採用
- ページ全体の高さやスクロール領域を作り直す
  - 変更範囲が広く、既存の編集画面へ影響するため不採用
- サイドバーへ画面高だけを指定する
  - 編集エリアが画面高を超えた場合に、同じ問題が残るため不十分

### 確認

- `npm run build` 成功
- TypeScript検査成功
- 開発サーバーの再コンパイルとHTTP 200応答を確認

## 2026-06-22: ToDo削除前の確認を追加

### 何を変更したか

- 個別ToDoの削除前に確認モーダルを表示
- 個別ToDoリストの削除前に、リスト名と配下タスク件数を表示
- 選択した複数ToDoリストの削除前に、リスト件数と配下タスク件数を表示
- `components/TodoDeleteConfirmModal.tsx` を追加し、削除確認UIを共通化
- `app/page.tsx` で削除要求を一度受け取り、確認後に既存の削除関数を実行

### なぜ変更したか

ToDoやToDoリストの削除は元に戻せず、特にリスト削除では配下のタスクも同時に失われます。
メモ削除と同様に確認を挟み、意図しないデータ消失を防ぐ必要がありました。

### なぜこの方法を選んだか

個別タスク、個別リスト、複数リストの削除要求を `page.tsx` に集約しました。
削除対象の状態と確認モーダルを1か所で管理でき、既存の `useTodos` の削除処理を変更せず再利用できるためです。
確認UIは専用コンポーネントへ分け、メモ削除と同じ配色・操作順を保ちました。

### 他に検討した方法

- `window.confirm` を使う
  - 実装は簡単ですが、ブラウザごとに見た目が異なり、アプリのUIと合わないため不採用
- `TodoPanel` と `TodoSidebar` で別々に確認状態を持つ
  - 同じ確認処理が分散し、文言やデザインがずれやすいため不採用
- 削除後に元へ戻す通知を表示する
  - 操作性は良いものの、削除データの一時保持が必要で今回の変更範囲を超えるため今後の候補

### 確認

- `npm run build` 成功
- TypeScript検査成功
- 個別ToDo、個別リスト、複数リストで異なる説明文を生成することをコード上で確認

## 2026-06-22: ToDoの選択削除を追加

### 何を変更したか

- 一括操作メニューへ `ToDoを選択して削除` を追加
- 選択モード中は各タスクのチェック欄を選択チェックとして使用
- 選択件数、削除、解除の操作をフィルター横へ表示
- フィルターを切り替えても、同じリスト内の選択状態を保持
- `useTodos` に複数ToDoをまとめて削除する `deleteTodos` を追加
- 複数ToDo削除時の確認モーダルに選択件数を表示

### なぜ変更したか

完了状態に関係なく、不要な複数タスクをまとめて削除する手段がありませんでした。
1件ずつ削除する負担を減らしつつ、確認画面によって誤削除を防ぐ必要がありました。

### なぜこの方法を選んだか

通常画面へ新しいボタンを常時追加せず、既存の一括操作メニューから選択モードを開始する方法を選びました。
選択中は既存チェック欄を再利用することで、行内の情報量とタップ領域を増やさずに済みます。
選択IDは表示中の配列ではなくリスト全体を基準に管理し、フィルター変更後も保持できるようにしました。

### 他に検討した方法

- 各行へ選択用チェック欄を常時追加する
  - 完了チェックと並んで用途が分かりにくくなり、通常時のUIが混雑するため不採用
- 表示中のタスクだけを選択対象として管理する
  - フィルター変更時に選択が消え、複数条件をまたいだ削除ができないため不採用
- 完了済み一括削除だけを使う
  - 未完了を含む任意のタスクを選べないため、別機能として選択削除を追加

### 確認

- `npm run build` 成功
- TypeScript検査成功
- 個別削除と複数削除が同じ確認モーダルを使用することをコード上で確認
- リスト変更時に選択モードと選択IDをリセットすることを確認
