# 技術概要と採用理由

この資料は、Simple Notesで使っている主要技術について、役割と採用理由を学習者向けにまとめたものです。
ライブラリの機能を網羅するのではなく、このプロジェクトのどこで役立っているかを中心に説明します。

## Next.js

### 概要

Next.jsは、Reactを使ってWebアプリを構築するためのフレームワークです。このプロジェクトではApp Routerを使い、`app/`ディレクトリを起点にページと共通レイアウトを構成しています。

### 採用理由

Reactだけでも画面は作れますが、ページ構成、開発サーバー、本番用ビルドなどを個別に整える必要があります。Next.jsを使うことで、アプリの基本構成と実行環境をまとめて用意でき、機能開発へ集中できます。また、Vercelへ公開しやすいことも採用理由です。

### 学習ポイント

- App Routerの基本構成
- `page.tsx`と`layout.tsx`の役割
- クライアントコンポーネントの扱い
- 開発環境と本番ビルドの違い

## React

### 概要

Reactは、画面をコンポーネントという部品に分け、状態に応じて表示を更新するためのUIライブラリです。このプロジェクトでは、サイドバー、メモ編集画面、ToDo画面、確認モーダルなどをコンポーネントとして構成しています。

### 採用理由

メモの選択、検索、テーマ切り替え、ToDoの完了状態など、操作によって変化する画面が多いためです。Reactを使うことで、DOMを直接書き換えず、現在の状態から画面を組み立てられます。共通UIを再利用しやすい点も、このアプリに合っています。

### 学習ポイント

- コンポーネントの分割と再利用
- Propsによるデータと操作関数の受け渡し
- `useState`による状態管理
- `useEffect`による保存や初期化
- カスタムフックによるUIとデータ処理の分離

## TypeScript

### 概要

TypeScriptは、JavaScriptに型の仕組みを加えた言語です。このプロジェクトでは、`Note`、`TodoList`、`Todo`などのデータ構造や、コンポーネントが受け取るPropsを型で定義しています。

### 採用理由

メモ、ToDoリスト、タスクは似た情報を持ちますが、それぞれ役割が異なります。型を使うことで、存在しない値の参照や、別のデータを誤って渡す問題を実行前に見つけやすくなります。機能追加時に「どのデータを変更する必要があるか」を追いやすい点も重要です。

### 学習ポイント

- オブジェクトと配列の型定義
- Propsと関数の引数・戻り値の型
- 文字列リテラル型による状態の制限
- localStorageから読み込んだ不明なデータの検証
- 型チェックを使った安全な変更

## Tailwind CSS

### 概要

Tailwind CSSは、短いユーティリティクラスを組み合わせてスタイルを作るCSSフレームワークです。このプロジェクトでは、余白、配色、レイアウト、レスポンシブ表示、ダークテーマなどに使っています。

### 採用理由

メモとToDoで似たUIを作りながら、PCとスマートフォン、ライトとダークの表示へ対応する必要があるためです。コンポーネントの近くでスタイルを確認でき、状態に応じたクラスの切り替えもReactと組み合わせやすいという利点があります。

### 学習ポイント

- FlexboxやGridを使ったレイアウト
- 余白、文字、境界線、背景色の設計
- ブレークポイントによるレスポンシブ対応
- 状態に応じたクラスの切り替え
- ライト・ダーク両方で読みやすいUI設計

## Tiptap

### 概要

Tiptapは、Web上でリッチテキストエディタを作るためのライブラリです。このプロジェクトでは、メモ本文の見出し、リスト、引用、コードブロック、Markdownショートカットなどを扱います。

### 採用理由

通常の`textarea`だけでは、文章の一部を見出しやリストとして構造化して表示できません。一方、編集履歴やカーソル位置を含むエディタ処理をすべて自作するのは複雑です。実績のある編集基盤を利用し、メモアプリ固有のUIや保存処理の学習へ集中するために採用しています。

### 学習ポイント

- 外部ライブラリをReactコンポーネントへ組み込む方法
- エディタの内容とアプリの状態を同期する方法
- 拡張機能によるエディタ機能の追加
- HTMLやリッチテキストを保存・復元するときの考え方
- 自作する範囲とライブラリへ任せる範囲の判断

## localStorage

### 概要

localStorageは、ブラウザ内に文字列データを保存する仕組みです。このプロジェクトでは、メモ、ToDo、タグ、ユーザー名、テーマ設定を保存し、再読み込み後も復元できるようにしています。

### 採用理由

サーバーやデータベースを準備せず、ブラウザだけで保存処理を学べるためです。JSONへの変換、読み込み時の検証、自動保存など、フロントエンドにおけるデータ管理の基本を小さな構成で経験できます。

ただし、別の端末とは同期されず、ブラウザのサイトデータを削除すると失われる可能性があります。本格的な永続保存ではなく、学習用アプリに適した選択です。

### 学習ポイント

- オブジェクトとJSONの相互変換
- 保存と読み込みのタイミング
- 古い形式や不正なデータの検証・正規化
- 状態変更と自動保存の連携
- ブラウザ保存の利点と制約

## Vercel

### 概要

Vercelは、Webアプリをインターネットへ公開するためのホスティングサービスです。このプロジェクトでは、GitHubのリポジトリと連携してNext.jsアプリを公開しています。

### 採用理由

Next.jsとの相性がよく、サーバーを自分で管理せずにビルドと公開を行えるためです。GitHubへpushした変更を公開サイトへ反映でき、ローカル開発から本番公開までの流れを小さな負担で経験できます。

### 学習ポイント

- ローカル環境と公開環境の違い
- GitHubとデプロイの連携
- 本番ビルドが成功することの重要性
- 公開URLでPC・スマートフォンから確認する流れ
- フロントエンドアプリを継続的に公開する基本

## このプロジェクト全体の学習目的

### なぜバックエンドを採用していないのか

最初からログイン、API、データベースまで導入すると、学ぶ範囲と不具合の原因が一気に増えます。このプロジェクトでは、まず画面設計、状態管理、型、安全なデータ操作といったフロントエンドの基礎へ集中するため、バックエンドを採用していません。

バックエンドが不要という意味ではありません。端末間同期、ユーザー認証、確実なバックアップが必要になった段階で、次の学習対象として検討できます。

### なぜlocalStorageを使っているのか

localStorageなら、ブラウザだけで「入力したデータを保存し、次回表示時に復元する」という一連の流れを実装できます。データベースを使わなくても、保存形式、読み込み、検証、互換性、自動保存という重要な考え方を学べるためです。

制約が分かりやすいことも学習上の利点です。どこまでlocalStorageで対応でき、どこからサーバー保存が必要になるかを比較できます。

### このアプリで身につけたいフロントエンド技術

- UIをコンポーネントへ分け、責務を整理する力
- React Hooksを使って状態と副作用を管理する力
- TypeScriptでデータ構造と操作を安全に設計する力
- 検索、絞り込み、並び替えなどの配列処理
- localStorageへ安全に保存・復元する力
- PC、スマートフォン、ライト、ダークに対応するUI設計
- 外部ライブラリを必要な範囲で導入する判断力
- GitHubとVercelを使って変更を公開する流れ

このプロジェクトの目的は、機能数を増やすことだけではありません。小さなアプリを継続して改善しながら、変更しやすい構造、安全なデータ処理、利用者に分かりやすいUIを考えられるようになることです。
