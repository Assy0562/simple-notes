# AGENTS.md

このファイルは、CodexなどのAIがこのプロジェクトで作業するときの共通ルールです。
人間向けの使い方は `README.md`、現在地は `docs/DEVELOPMENT_STATUS.md`、構造は `docs/PROJECT_STRUCTURE.md` を参照してください。

## プロジェクト方針

- 学習用の個人開発として、分かりやすさと保守性を優先する
- Next.js App Router、TypeScript、Tailwind CSSを使う
- 既存の構造と実装パターンを優先し、必要のないライブラリを増やさない
- 一度に広範囲を変更せず、依頼された機能へ変更範囲を絞る
- UIはメモとToDoで一貫させ、ライト・ダーク・PC・スマートフォンを考慮する

## 作業前

1. `docs/DEVELOPMENT_STATUS.md`で現在地を確認する
2. `git status -sb`でユーザーの未コミット変更を確認する
3. 関係するファイルだけを読む
4. 実装前に、変更対象と方針を日本語で簡潔に説明する

不明点はコードから確認し、挙動やデータ消失に関わる判断だけユーザーへ質問します。

## 実装

- UIは `components/`、状態と保存は `hooks/`、データ変換は `lib/`、型は `types/` の責務を守る
- localStorageの既存キーや保存形式を変更するときは、既存データの互換性を確認する
- データ構造を変更するときは、型、初期データ、読み込み時の正規化をセットで確認する
- 削除や初期化など元に戻せない操作には確認を設ける
- 初心者向けコメントは、処理の理由がコードだけでは分かりにくい場所に限定する
- 無関係な整形、リネーム、リファクタリングを混ぜない
- ユーザーの既存変更を勝手に戻さない

## ドキュメント

- 利用者向け機能や起動方法が変わる場合は `README.md` を更新する
- ファイル追加や責務変更がある場合は `docs/PROJECT_STRUCTURE.md` を更新する
- 開発の現在地、注意点、次候補が変わる場合は `docs/DEVELOPMENT_STATUS.md` を更新する
- 判断を伴う機能追加や不具合修正は `docs/IMPLEMENTATION_NOTES.md` に記録する
- 文言、余白、単純なアイコン調整などは、関連する記録へまとめられる場合は個別項目を増やさない

実装記録には原則として次を残します。

- 何を変更したか
- なぜ変更したか
- なぜその方法を選んだか
- 他に検討した方法
- 確認方法と結果

## 確認

最低限:

```bash
npm run typecheck
```

リリース前、共通処理、保存形式、依存関係へ影響する変更:

```bash
npm run build
```

UI変更は可能な範囲でブラウザでも確認します。確認できなかった項目は、完了報告で明示します。

## Git

- `.next/`、`node_modules/`、`tsconfig.tsbuildinfo`、ログをコミットしない
- `.vscode/`は、共有する内容をユーザーが明示した場合だけコミットする
- 変更ファイルを確認し、今回の作業に関係するファイルだけをステージする
- push、merge、ブランチ変更はユーザーの依頼がある場合に行う

## 完了報告

実装後は日本語で簡潔に次を報告します。

- 変更した内容
- 主な判断理由
- 実行した確認
- 未確認または残っている課題
